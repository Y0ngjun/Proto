# Copilot 작업 워크플로우

사용자 요청 처리 시 따라야 할 **4단계 절차** + **검수 단계**입니다.

> 자세한 코딩 원칙은 `.github/copilot-instructions.md` 참조

---

## 📋 전체 흐름

```
단계 1-2 → 검수 대기 → 단계 3-4
  ↓         (사용자)      ↓
[계획 수립]           [구현 실행]
```

---

## Phase A: 계획 수립 (단계 1-2)

### 단계 1: 문서 읽기

- [ ] `.github/copilot-instructions.md` (중앙 지침)
- [ ] `docs/style-guide.md` (코드 스타일)
- [ ] `docs/testing.md` (테스트 방법)
- [ ] `docs/progress.md` (현재 상태)
- [ ] 관련 `.md` 파일 (`src/Core.md`, `src/Scene.md` 등)

**팁**: 파일/클래스명이 명확하면 해당 디렉토리 md만 읽으면 됨

### 단계 2: 계획 수립

- [ ] 요구사항 명확히 이해 (모호하면 질문)
- [ ] 영향 범위 파악 (수정 파일 목록)
- [ ] 구현 순서 결정
- [ ] 성공 기준 정의

**제공 내용**:
- 📊 현재 상태 분석
- 🎯 구체적인 계획 제시
- 📝 수정할 파일 목록
- ✅ 성공 기준 명시

---

## 🔍 검수 포인트 (사용자 확인)

**Copilot이 제시한 계획 검토:**

- [ ] 요구사항을 정확히 이해했는가?
- [ ] 영향 범위가 적절한가?
- [ ] 구현 순서가 합리적인가?
- [ ] 성공 기준이 명확한가?
- [ ] 다른 방법이 더 나은가?

**피드백 방법:**
- ✅ "진행해줘" → Phase B 실행
- 🔄 "다시 계획해줘" → Phase A 반복
- 📝 "이 부분 수정해줘" → 피드백 반영 후 진행

---

## Phase B: 구현 실행 (단계 3-4)

*검수 통과 후 실행*

### 단계 3: 작업 실행

- [ ] 필요한 파일 읽기 (코드 컨텍스트 확보)
- [ ] 코드 수정 (`replace_string_in_file` / `multi_replace_string_in_file`)
- [ ] 빌드 검증 (`run_build`)
- [ ] 테스트 실행 (`run_tests`)

### 단계 4: 문서 업데이트

- [ ] `docs/progress.md` - 완료 작업 표시, Phase 업데이트
- [ ] 관련 `.md` - 새 클래스/메서드 추가 시 반영
- [ ] `docs/spec/architecture.md` - 구조 변경 시에만 수정

---

## 작업 타입별 체크

| 타입 | 절차 |
|------|------|
| 🐛 **버그** | [Phase A] 원인 분석, 최소 수정 계획 → 검수 → [Phase B] 수정, 테스트, 기록 |
| ✨ **기능** | [Phase A] 요구사항 명확, 위치 결정, 구현 계획 → 검수 → [Phase B] 구현, 테스트, 문서 |
| 🔄 **리팩** | [Phase A] 범위 확정, 안전성 검토 → 검수 → [Phase B] 수정, 회귀 테스트 |
| 📚 **문서** | [Phase A] 대상 파악 → 검수 → [Phase B] 작성, 유지 |

---
